## Суть изменений

PR добавляет SQL export для twin factories — генерацию INSERT statements для переноса конфигурации factories между средами. Реализация следует паттерну уже существующего `TwinClassExportService`.

**Структура:**
- 1 новый controller (`FactoryExportSqlController`)
- 1 новый DTO (`FactoryExportSqlRqDTOv1`)
- 1 оркестрирующий сервис (`FactoryExportService`)
- 5 суб-экспорт сервисов (Branch, Multiplier, Pipeline, Eraser, Trigger)
- 1 разделяемый сервис (`FactoryConditionSetExportService`)
- 9 batch-методов в репозиториях
- 8 proxy-методов в сервисах

**26 файлов, +654 строки, 0 тестов.**

## Раунд 1 — Разведка (независимые позиции)

### Марина «Скальпель» — ТРЕБУЮТСЯ ИСПРАВЛЕНИЯ

**Позиция:** Найден функциональный баг — дублирование SQL INSERT при комбинированном экспорте. Если conditionSet используется branch и eraser одновременно, генерируются два INSERT для одной строки.

**Найденные антипаттерны:**
- АП-1: Копипаст Branch/Eraser/Trigger export (80% идентичного кода)
- АП-2: 5x `new ArrayList<>(factoryIds)` — повторяющееся преобразование
- АП-3 (КРИТИЧЕСКИЙ): Дублирование conditionSetIds при комбинированном экспорте — duplicate key violation при импорте
- АП-4: Нет валидации `twinFactoryIds` (null → NPE)
- АП-5: `findByTwinFactoryIdIn` не фильтрует по `active` (расхождение с single-ID методами)
- АП-6: 0 тестов при 654 новых строках

**Расхождения с эталоном:**
- `@EqualsAndHashCode(callSuper = false)` вместо `true`
- `findByTwinFactoryId(UUID)` добавлен, но нигде не вызывается — мёртвый код

**Ключевой вопрос:** Шарятся ли conditionSets между branch/eraser/trigger? Если да — АП-3 это баг.

### Тимур «Индекс» — REQUEST CHANGES

**Позиция:** 45-65 SQL-запросов при экспорте 10 factories с 5 флагами — из которых ~30-40 дублирующиеся. При правильной дедупликации — 12-15 запросов. Плюс 2 отсутствующих индекса.

**Найденные антипаттерны:**
1. [КРИТИЧЕСКИЙ] Дублирующаяся загрузка ConditionSets — ~30-40 лишних запросов
2. [КРИТИЧЕСКИЙ] N+1: `findByTwinFactoryId(UUID)` вместо batch — 10 запросов вместо 1
3. [ВЫСОКИЙ] Отсутствующие индексы:
   - `twin_factory_condition_set.twin_factory_id` — **Seq Scan**
   - `twin_factory_pipeline_step.twin_factory_pipeline_id` — **Seq Scan**
4. [СРЕДНИЙ] Нет `@Transactional(readOnly = true)`
5. [СРЕДНИЙ] 5x `new ArrayList<>(factoryIds)`
6. [НИЗКИЙ] Нет batch size limit для IN-списков

**Уникальные находки:**
- Два отсутствующих индекса — реальный performance impact
- `findByTwinFactoryId(UUID)` — N+1 паттерн (принимает одиночный UUID вместо Collection)

### Олег «Замок» — НЕ РЕКОМЕНДУЕТ МЕРДЖИТЬ

**Позиция:** 3 критических security-проблемы: отсутствие input validation (DoS вектор), отсутствие audit logging (compliance), error oracle (information leakage).

**STRIDE Threat Modeling:**
- S (Spoofing): НИЗКИЙ — `@ProtectedBy` + `AuthService` корректны
- T (Tampering): НИЗКИЙ — read-only операция
- R (Repudiation): **ВЫСОКИЙ** — нет audit trail (GDPR, SOC 2, ISO 27001)
- I (Information Disclosure): **СРЕДНИЙ** — error oracle через `UUID_UNKNOWN` vs `ACCESS_DENIED`
- D (Denial of Service): **ВЫСОКИЙ** — нет лимита на количество UUID
- E (Elevation of Privilege): НИЗКИЙ — `FACTORY_MANAGE` корректен

**Найденные антипаттерны:**
1. [КРИТИЧЕСКИЙ] Нет input validation — OWASP API4:2023 Unrestricted Resource Consumption
2. [ВЫСОКИЙ] Нет audit logging — кто, когда, что экспортировал — неизвестно
3. [СРЕДНИЙ] Error oracle — `findEntitiesSafe()` выдаёт `UUID_UNKNOWN` вместо generic ошибки
4. [СРЕДНИЙ] Permission granularity — export доступен только `FACTORY_MANAGE`, не `FACTORY_VIEW`
5. [НИЗКИЙ] Content-type `text/sql` — нестандартный MIME type
6. [НИЗКИЙ] Предсказуемый filename (timestamp)

**Позитивные находки:**
- Domain isolation через `isEntityReadDenied()` — корректна
- `BulkheadFilter` — базовый rate limiting
- `SqlBuilder.escapeSqlString()` — экранирование
- `ON CONFLICT DO NOTHING` — безопасный паттерн импорта

## Раунд 2 — Столкновение

### Повестка (от Навигатора)
1. Является ли дублирование INSERT функциональным багом или допустимым?
2. Блокирует ли копипаст мерж или можно отложить?
3. Нужно ли добавлять validation, если эталон её не имеет?

### Дискуссия

**Марина:** "Дублирование INSERT — это баг. При комбинированном экспорте `includeBranches=true + includeErasers=true` один conditionSet экспортируется дважды. Если SqlBuilder генерирует `INSERT ... ON CONFLICT DO NOTHING` — это молчаливый пропуск. Но если целевая БД без constraint — будет duplicate rows. Это неопределённое поведение."

**Тимур:** "Подтверждаю. При 10 factories с 5 флагами — 45-65 SQL-запросов, из которых 30-40 дублирующиеся. Это не теоретический риск — это реальное ухудшение производительности. Плюс результат — раздутый SQL-файл с повторяющимися INSERT."

**Игорь:** "Согласен, что это нужно чинить, ноON CONFLICT DO NOTHING в SqlBuilder — это стандартный паттерн проекта. Дублирующиеся INSERT не вызовут ошибку. Проблема в раздутом SQL, не в валидности. Это P1, не P0."

**Дмитрий:** "Не согласен с Игорём. Копипаст + дублирование — это не P1, это P0. Потому что багфикс в conditionSet-экспорте потребует правок в 4 местах. Это maintenance nightmare. Лучше устранить сейчас, пока код не в проде."

**Марина:** "Поддерживаю Дмитрия. Три копии 15-строчного блока — это ровно та ситуация, когда рефакторинг дешев сейчас и дорог потом."

**Олег:** "Добавлю к validation — отсутствие `@NotEmpty` на `twinFactoryIds` это не просто style issue. Это OWASP API4:2023. При `null` — NPE (500), при empty set — 200 с пустым файлом. Оба сценария — неправильное поведение. Три строчки кода (`@NotEmpty` + `@Valid` + `@Size`)."

**Тимур:** "Отдельно хочу подсветить отсутствующие индексы. `twin_factory_condition_set.twin_factory_id` и `twin_factory_pipeline_step.twin_factory_pipeline_id` — без индексов каждый batch-запрос будет Seq Scan. Сейчас таблицы маленькие, но при росте это будет проблемой."

**Игорь:** "Согласен с индексами — это нужно добавить в миграцию. Это не относится напрямую к PR, но раз уж добавляем batch-методы по этим колонкам — индексы логично добавить здесь же."

### Статус разногласий (итог Навигатора)

- **Консенсус достигнут:**
  - Дублирование conditionSets нужно устранить (P0)
  - Копипаст нужно устранить через обобщённый метод (P0)
  - `@EqualsAndHashCode(callSuper = true)` — исправить (P0)
  - 5x `new ArrayList` — вынести на уровень метода (P1)
  - `@Transactional(readOnly = true)` — добавить (P1)
  - Мёртвый код `findByTwinFactoryId` — удалить (P1)
  - 2 индекса — добавить миграцию (P1)

- **Раскол сохраняется:**
  - Олег: validation — P0 (DoS вектор). Игорь: P2 (унаследовано). Консенсус: P1 компромисс — `@NotEmpty` без `@Size`.

- **Новые вопросы:**
  - Нужно ли добавлять audit logging сейчас или в отдельном тикете?
  - Нужно ли `findByTwinFactoryIdIn(Collection)` вместо singleton?

## Синтез

### Ключевые выводы

1. **Архитектурно код корректен** — следует проверенному паттерну TwinClassExportService, использует batch-загрузку вместо N+1, корректно разделяет ответственности между оркестратором и суб-сервисами.

### Инсайты

1. **Тимур нашёл 2 отсутствующих индекса**, которые никто другой не заметил — это уникальная ценность DBA-экспертизы в ревью. Seq Scan на `twin_factory_pipeline_step` будет ухудшаться с ростом данных.

2. **Марина нашла мёртвый код** `findByTwinFactoryId(UUID)` — метод добавлен, но нигде не вызывается. Это сигнал о незавершённом рефакторинге: планировался batch-запрос, но забыли/не успели.

3. **Копипаст в Branch/Eraser/Trigger — не просто style issue.** Если завтра обнаружится баг в условии `entity.getTwinFactoryConditionSetId() != null` — его придёт фиксить в 3 местах. Это классический риск дублирования.
