# I18n

## Summary

A translation-key entity. Multi-language display strings across the platform are stored as UUID references to I18n rows, with the actual per-locale text living in `I18nTranslation` rows. I18n itself carries the key, type, and domain scoping; translations carry the values.

## Purpose

Twins is multi-locale by design — every Domain has a default locale and can support many. Rather than storing raw display strings on each entity (TwinClass name, TwinStatus description, Permission label, …), the platform stores a UUID reference to an I18n row. The I18n row is the translation key; the matching `I18nTranslation` rows carry one value per locale.

This split has three benefits. (1) Adding a new locale is non-breaking — just add new translations, no schema changes. (2) Cache-friendly — the platform can cache I18n rows separately from translations and assemble them at render time. (3) Type-safe — every entity that has a name has a `nameI18nId` UUID, regardless of whether the name is short text, styled text, image alt-text, or external ID.

The `type` enum (`I18nType`) classifies what the key is for (`TWIN_CLASS_NAME`, `TWIN_STATUS_DESCRIPTION`, `PERMISSION_NAME`, … — ~40 values). The platform uses the type to route validation (e.g., TWIN_CLASS_NAME translations cannot be empty), drive UI presentation (text vs styled-text vs image), and group translations for bulk operations. Each type also carries a `Category` (`TEXT` / `STYLED_TEXT` / `IMAGE` / `EXTERNAL_ID`) that drives higher-level handling.

I18n rows are scoped to a Domain (`domainId`). Same key in different Domains produces different I18n rows — Domains do not share translation keys.

## Fields

- `id` — primary key (UUID v7)
- `domainId` — FK to [Domain](domain.md), owner domain
- `key` — short identifier within the domain (e.g., `issue.class.name`, `permission.TWIN_CREATE.name`)
- `name` — human-readable label for the key itself (admin UI hint)
- `type` — enum `I18nType` (converted via `I18nTypeConverter`) — ~40 values

## API

I18n itself:

- `GET    /private/i18n/{i18nId}/v1` — view single I18n key

Translations (the actual per-locale values):

- `POST   /private/i18n_translation/search/v1` — search translations for given I18n ids
- `PUT    /private/i18n_translation/v1` — create / update translations (batch upsert)

The platform does not expose create / delete endpoints for I18n itself — keys are created implicitly when an entity references a new i18n_id UUID, and removed via cascade when the owning entity is deleted.

## Examples

A typical I18n key for a TwinClass name:

```json
{
  "id": "0192a7c4-...",
  "domainId": "0192a7b0-...",
  "key": "issue.class.name",
  "name": "Issue class display name",
  "type": "TWIN_CLASS_NAME"
}
```

Translations (separate rows in `i18n_translation`):

```json
[
  { "i18nId": "0192a7c4-...", "locale": "en", "value": "Issue" },
  { "i18nId": "0192a7c4-...", "locale": "ru", "value": "Задача" },
  { "i18nId": "0192a7c4-...", "locale": "es", "value": "Incidencia" }
]
```

When a client renders the parent TwinClass with locale=`ru`, the platform resolves via `I18nService` and returns "Задача".

## Dev notes

- **Never store raw translatable strings on entities** — always use `*I18nId` UUID columns. Storing raw strings breaks i18n and causes silent data divergence between locales.
- **`I18nService.collectI18nIds(...)` is the canonical way** to collect i18n UUIDs from a collection of entities — never iterate manually. It handles `null` checks and field selection uniformly.
- **I18n is created implicitly** — when an entity references a new UUID in its `*I18nId` column, an I18n row should exist (typically pre-created via Flyway or service-layer logic). Missing rows cause translation lookup failures.
- **Deletion is cascaded** — when the owning entity is deleted, the matching I18n rows are removed by DB cascade. Do not delete I18n rows directly.
- **`type` drives validation and UI presentation** — `TWIN_CLASS_NAME` cannot have an empty translation; `IMAGE` translations are URLs; `STYLED_TEXT` allows HTML. Always set the correct type for new keys. Full list (~40 values): [I18nType.java](../../../../core/src/main/java/org/twins/core/enums/i18n/I18nType.java).
- **For SQL export ordering** — when exporting entities via the Export API, I18n rows must be exported BEFORE the entities that reference them (FK constraint order). `I18nExportService` handles this.
